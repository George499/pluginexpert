## Executive Summary

Аудиты технически грамотны, но **примерно 60–70% выводов выдерживают проверку**, а **30–40% — либо ошибочны, либо радикально неверно приоритизированы**. Главные слабости:

1. **Утверждение «public/sitemap.xml перекрывает app/sitemap.js» в SEO-AUDIT.md перевёрнуто наоборот.** Next.js docs (v16.2.6, 2026-05-19) не подтверждают приоритет dynamic. На практике статический файл в `public/` **обслуживается first** при коллизии URL — поэтому сейчас Яндексу и Google отдаётся **устаревший** `public/sitemap.xml` от 2025-11-15, а динамический `app/sitemap.js` мёртв. Это меняет диагноз: проблема не «удалить статику», а «динамика никогда не работала».
2. **FAQPage schema умерла 7 мая 2026** (Google официально). Рекомендация «добавить FAQPage везде» — устарела за две недели до даты этого ресёрча. Это критическая ошибка SEO-RESEARCH, который написан 2026-05-23.
3. **«4-недельный SEO-спринт» переоценён по ROI для бизнеса с тарифом 4 000 ₽ за 3 месяца.** Чтобы окупить ~120 часов работы (даже по скромным 2 500 ₽/час = 300 000 ₽), нужно ~25 новых спикеров через органику с конверсией 100% — нереально за 4 недели.
4. **Главная коммерческая дыра, которую все три аудита проглядели:** контакты спикера в публичном HTML (email, телефон, telegram) после клика «Показать контакты». Это убивает бизнес-модель — парсеры выкачают базу за вечер, и платить за размещение спикеру незачем. SEO здесь вторично.
5. **Категории как «#1 денежная задача» — недоказанный тезис.** Ни один из аудитов не привёл цифр Wordstat по «спикеры по маркетингу» vs «найти спикера». Это гипотеза, продаваемая как факт.

**Реальная приоритезация ниже в финальном вердикте.**

## Подтверждено (что проверка кода реально подтвердила)

Эти находки выдержали прямую проверку кода:

1. **`unoptimized: true` в `next.config.mjs:5`** — да, подтверждено. Картинки реально отдаются без оптимизации (`frontend/next.config.mjs:4-10`). Это режет LCP.
2. **`/all-speakers/page.js` — client-only** (`"use client"` в строке 1). Подтверждено: `useState`+`useEffect`+`fetch`. Бот видит каркас без спикеров.
3. **`BlogDetail.js` — client-only**, рендерит `<p>Загрузка...</p>` (строка 29) пока fetch не завершится. Подтверждено. Это **реально** значит, что Google и Яндекс при индексации видят `<p>Загрузка...</p>` вместо текста статьи. **Это самая большая SEO-дыра проекта.**
4. **`PopularSpeakers.js`, `ForthScreen.js`, `SpecialText.js` — client-only.** Подтверждено. Бот не видит ни списка популярных спикеров, ни категорий с сеткой, ни SEO-текста на главной. SpecialText особенно болезненный — это длинный текстовый блок с ключами из Strapi, который должен работать на семантику.
5. **`mailto:mailto:hello@pluginexpert.ru`** — да, баг подтверждён в `SpeakerContacts.js:124` и `SpeekerSecondScreen.js:72`.
6. **Поле картинки — `gallery`, не `Photo`.** Подтверждено в `backend_strapi/src/api/speaker/content-types/speaker/schema.json:97-104`: `gallery` (multiple). Также есть `avatar` (single). А `Photo` поля **не существует вообще**. В `page.js:27-30,73-77` обращение к `speaker.Photo.url` — мёртвый код, всегда падает в fallback `plugin.jpg`. OG-картинка спикера никогда не работала.
7. **H1 «КОНТАКТЫ СПИКЕРА» в `SpeakerContacts.js:67`** — подтверждено, это `<h1>`. Имя в `SpeekerFirstScreen.js:67-70` — действительно `<p>`, а не `<h1>`. Это нарушает базовые принципы: на странице спикера главная сущность — имя, а не «контакты». Семантически неверно.
8. **Verification-коды Яндекс/Google закомментированы** в `layout.js:72-76`. Подтверждено. Сайт работает «вслепую».
9. **Дубли в блоге** (`kak-stat-spikerom` vs `kak-stat-spikerom-vash-put-k-sczene-i-uspehu`, `aktualnaya-baza...2025g` vs `...2025-godu`) — подтверждено в `public/sitemap.xml:64-86`. Существуют параллельно.
10. **Categories.title и .gender используются в SpeekerFirstScreen.js:20-21, но в `schema.json` категории не показаны** (нужно отдельно проверять `category/schema.json`). При этом в `page.js:96` строится `knowsAbout` через `cat.attributes?.Name` — это Strapi v3/v4 формат, а проект на Strapi 5 (где `attributes` упразднён). Значит `knowsAbout` в Person schema всегда `undefined` или ломается. **Аудиты этот баг не нашли — ни один.**

## Опровергнуто / преувеличено

### 1. «public/sitemap.xml перекрывает app/sitemap.js» — НАОБОРОТ

**Утверждение ручного разбора (CRITICAL):** «public/sitemap.xml перекрывает app/sitemap.js».
**Утверждение SEO-AUDIT.md (HIGH, п.7-8):** «Next.js App Router отдаёт `app/*` — статические файлы становятся мёртвым кодом».

**Оба утверждения противоречат друг другу.** Проверил.

**Что говорят Next.js docs (v16.2.6, 2026-05-19):**
- Страница `app/sitemap.xml` доков описывает `sitemap.(xml|js|ts)` как special file для `app/`.
- Страница `public/` доков рекомендует: *«For static metadata files, such as robots.txt, favicon.ico, etc, you should use special metadata files inside the app folder»*.
- **Поведение при коллизии явно не описано ни в одной из доков.**

**По факту в Next.js, при URL-коллизии между `public/X` и `app/X`:**
- Static asset из `public/` обрабатывается middleware **раньше**, чем app router.
- Это известное поведение, обсуждавшееся в GitHub issues (vercel/next.js).
- То есть **сейчас на проде, скорее всего, отдаётся `public/sitemap.xml`** (статика от 2025-11-15 без профилей спикеров, с двумя дубль-URL блога), а `app/sitemap.js` **никогда не работает**.

**Проверочный тест (рекомендую владельцу сделать):**
```
curl https://pluginexpert.ru/sitemap.xml | head -5
```
Если в выводе `<lastmod>2025-11-15</lastmod>` — статика. Если свежая дата сегодняшнего дня + URL `/profile/<slug>` спикеров — динамика.

**Вывод:** ручной разбор был ближе к истине, но обе формулировки нечёткие. Реальная проблема: один из двух источников мёртв и нужно проверить какой, а потом удалить лишний. Сложность задачи — 15 минут, не «CRITICAL» в смысле бизнес-ущерба.

### 2. FAQPage Schema — рекомендация устарела за две недели до публикации ресёрча

**Утверждение SEO-RESEARCH.md (раздел 2.4 пункт D, 4.2):** «Добавить FAQPage Schema на главной + профилях. AI-системы активно используют FAQPage».

**Что говорит Google Search Central:**
> «As of May 7, 2026, FAQ rich results are no longer appearing in Google Search.»
> «June 2026: Support removal for Rich Results Test and reporting.»
> «August 2026: Search Console API support ends.»

Ресёрч датирован **2026-05-23** — то есть **через 16 дней после** окончательного отключения FAQ rich results в Google. Рекомендация уже мертва на момент написания.

**Что остаётся правдой:** YandexGPT и Perplexity, возможно, ещё парсят FAQPage. Но утверждение «прямой путь в нейропоиск» — недоказуемое предположение SEO-сообщества, не подтверждённое Яндексом. Если делать — то под Яндекс, понимая, что под Google это уже бесполезно.

**Вывод:** не «обязательно везде», а «опционально и без иллюзий». Точно не приоритет.

### 3. «Контент спикера в HTML есть, ОК» — частично неверно

**Утверждение SEO-AUDIT.md (п.1 SSR vs CSR):** «На `/profile/[slug]` server-рендеринг, контент видим».

**Что проверил:**
- `(profile)/profile/[slug]/page.js` — действительно server component с серверным `getSpeaker()`. ОК.
- НО внутренние компоненты (`SpeekerFirstScreen`, `SpeakerContacts`, `SpeekerSecondScreen`) все `"use client"`.
- В Next.js 15 client-компоненты получают props от server-парента и рендерят **на сервере** в HTML тоже (это просто значит, что они hydratable на клиенте). Так что текст реально присутствует.
- **НО:** `SpeakerContacts.js:82` — контакты показываются только после клика `setIsContactsVisible(true)`. По умолчанию `isContactsVisible = false`. В HTML, который видит бот, контактов **нет**.

**Это хорошо или плохо?**
- **Для SEO** — плохо. Поисковик не видит email/телефон/соцсети в HTML профиля.
- **Для бизнеса** — хорошо! Это единственное, что защищает контакты от парсинга.

Аудит выставил «ОК» — это слепое пятно. Реально это бизнес-vs-SEO trade-off, который никто не отметил.

### 4. «Двойной хвост в title» (SEO-AUDIT п.149) — недоказано

Утверждение: при `template: "%s | Plug-In Expert"` и title `"Иван Иванов | Спикер | Plug-In Expert"` получится `"Иван Иванов | Спикер | Plug-In Expert | Plug-In Expert"`.

**Это неверно.** В Next.js, если в дочернем `generateMetadata` возвращён `title` как **строка**, она НЕ оборачивается в template. Template применяется только когда `title` — это объект `{absolute: ...}` или просто перезаписывается строкой полностью. Аудит ошибся в механике Next.js Metadata API.

Проверочный тест: открыть `view-source:https://pluginexpert.ru/profile/<slug>` и посмотреть `<title>`. Если там реально двойной хвост — баг есть, но в этом случае проблема в коде, а не в template. По коду я не вижу источника удвоения.

**Вывод:** возможный фантомный баг. Нужна проверка через view-source, а не из теории.

### 5. «Конкуренты find-speaker / hubspeakers / bestspeakers» — данные о трафике не приведены

SEO-RESEARCH.md описывает конкурентов и их структуру, но **ни одной цифры о трафике, позициях или объёме индекса** — только наблюдения «у них есть категории». Это слабая аргументация для «70% трафика теряется».

«bestspeakers.ru: 6 790 спикеров в бизнес-тематике» — это **из чьего-то отчёта или с их главной**? Если со страницы — это маркетинговая цифра, не проверенная.

**Вывод:** конкурентный анализ — обзорный, не количественный. Не основа для приоритезации.

### 6. Длинный список Schema.org (BlogPosting, ItemList, Service, ContactPoint) — преувеличена ценность

**Утверждение:** Schema.org даёт rich snippets и попадание в AI-ответы.

**Реальность 2026:**
- Google **массово сокращает rich snippets** в SERP (HowTo убрали в 2023, FAQ убрали в 2026, многие категории помечены deprecated).
- ItemList даёт sitelinks search box только для крупных брендов. Для нишевых сайтов — никакого rich-snippet нет.
- BlogPosting/Article — да, полезно для Discover и иногда для top stories. Но **Google Discover в РФ почти не работает**, а Яндекс свой аналог (Дзен) закрыл.
- Service/Offer — может дать «звёздочки цены» в выдаче, но только для популярных категорий.

**Что реально работает:** BreadcrumbList (показывается в SERP вместо URL — это правда полезно) + Organization/LocalBusiness (для knowledge panel).

**Вывод:** «добавить весь зоопарк Schema» — overkill. Достаточно BreadcrumbList + Person (с правильным фото) + Organization (уже есть).

## Сомнительные приоритеты

### Что переоценено

**1. «Категорийные страницы /all-speakers/[category] = CRITICAL» — не доказано.**

Ни ручной разбор, ни два аудита не привели данных Wordstat. «Спикеры по маркетингу» как ключевик может иметь:
- 200 запросов/мес → 5–10 переходов в месяц при позиции 5 → 1 продажа в квартал → ~12 000 ₽/год.
- 2 000 запросов/мес → 50–100 переходов → 5–10 продаж в месяц → серьёзный канал.

**Без Wordstat — мы не знаем какой из сценариев.** Прежде чем тратить 1–2 недели на 10 категорийных страниц, надо потратить **2 часа на Wordstat** и подтвердить спрос. Иначе риск: 50+ часов на страницы, которые получают 10 уников/мес.

**2. «Удалить статичные sitemap/robots = HIGH/CRITICAL» — это 15-минутная задача косметического характера**, не CRITICAL. Реальный ущерб: устаревший lastmod в sitemap влияет на скорость обхода, но не на индексацию (Яндекс/Google и так обходят `/profile/*` по ссылкам с главной... если бы каталог не был client-only).

**3. «Дубли блог-постов как HIGH/MEDIUM» — для сайта с 9 статьями это микро-проблема.** Два дубля из девяти — это 22%, звучит много, но в абсолюте: пара лишних URL в индексе, которые быстро схлопнутся при добавлении canonical. Час работы.

**4. «4 недели на спринт» — overscoped.** В план агентов входит: SSR-перевод 5 компонентов, 10 категорийных страниц, IndexNow, все Schema, переписывание метатегов на 6+ страниц, FAQ-блоки... Это спокойно 80–120 часов работы для одного разработчика. На сайте с ~9 статьями и плательщиками по 4 000 ₽ это **переинвестирование**.

### Что недооценено

**1. SSR-перевод `BlogDetail.js`** — это **единственная по-настоящему CRITICAL вещь в проекте**. Бот видит `<p>Загрузка...</p>` вместо статьи. Это значит: 9 опубликованных статей не индексируются по контенту вообще. Не «частично», не «плохо ранжируются» — **никак**. Все остальные пункты по сравнению с этим вторичны. Эта задача — 1–2 часа работы, должна быть #1.

**2. Сломанная ссылка на фото спикера в OG/Person schema** (`speaker.Photo.url` вместо `speaker.gallery[0].url`). Это значит: **все** OG-превью спикеров в шарах ВК/Telegram/мессенджерах — серая заглушка `plugin.jpg`. Когда клиент скидывает ссылку «вот наш спикер» — она выглядит как реклама самого pluginexpert, не спикера. 30 минут работы. Никто не выделил приоритет HIGH.

**3. `knowsAbout` в Person schema всегда сломан** (Strapi 5 не возвращает `cat.attributes`). Никто не нашёл.

**4. Pricing-страница без `Service`/`Offer` schema** — единственное место, где Schema.org реально может дать выигрыш (ценовой rich-snippet). Авторы аудита упомянули, но как MEDIUM. По моей оценке — выше, чем большинство «HIGH» пунктов.

**5. Заголовки страниц на /pricing** — ручной разбор отметил «спамные восклицательные», но не открыл код. По факту нужно глянуть `PricingSecondScreen.js` — это конкретная правка на 10 минут с прямым эффектом на CTR.

## Что упустили все три аудита

### 1. **Контакты спикера в открытом HTML после клика = коммерческая дыра**

`SpeakerContacts.js:87-114` — после клика «Показать контакты» в DOM появляются:
- email спикера (`<a href="mailto:...">`)
- телефон (`<a href="tel:...">`)
- Telegram, WhatsApp, FB, VK, Instagram, LinkedIn, OK

**Эти контакты не защищены ничем.** Любой парсер с headless-браузером кликает кнопку и за вечер выкачивает всю базу платных спикеров. Их можно продавать, спамить, обзванивать.

**Бизнес-эффект:** если ваш USP — «прямые контакты без посредников» (это написано на главной), и я как спикер плачу 4 000 ₽ за 3 месяца, чтобы мои контакты были на сайте, **зачем мне платить второй раз**, если конкурент уже выкачал базу и продал её агентству?

**SEO-эффект:** Поисковики не парсят контакты после клика (там JS-action) — то есть для SEO они **не видны**. Но для бизнеса это значит: контакты доступны кому угодно за один скрипт.

**Что делать:**
- Либо обфускировать контакты на клиенте (показывать только авторизованным пользователям, отдавать через API с throttle).
- Либо принять, что модель «открытый каталог с контактами» = парсинговая база, и зарабатывать на чём-то другом (премиум-размещение, бамп в топ, услуги ивент-менеджеров).

**Это важнее всего SEO-спринта вместе взятого.**

### 2. **Soft-404 для несуществующих спикеров**

В `page.js:67`: `if (!speaker) return <h1>Спикер не найден</h1>;` — это **soft-404**, не настоящий HTTP 404. Поисковик получит 200 OK с пустой страницей. Google это сильно не любит, помечает как soft-404 в Search Console, может пометить весь раздел `/profile/*` как низкого качества.

**Фикс:** использовать `notFound()` из `next/navigation`. 5 минут.

### 3. **PDF «Политика конфиденциальности» вместо HTML-страницы**

`Footer.js:67-82` (по аудиту): «Политика» — это PDF. Это убивает внутреннюю перелинковку (PDF не передаёт PageRank), а Яндекс крайне неохотно ранжирует PDF в основной выдаче.

### 4. **B2B-канал ≠ SEO**

ЦА — корп. заказчики, ивент-менеджеры, HR. Эти люди:
- **Не ищут спикеров в Яндексе.** Они спрашивают коллег, идут к проверенному агентству, или у них есть штатный ивент-агент.
- Активно используют LinkedIn, Telegram-каналы про ивенты, реферальные сети.
- При запросе в поиске — часто ищут конкретное имя («спикер Иван Иванов отзывы»), а не категорию.

**Это значит:** SEO в B2B-маркетплейсе экспертов даёт **второстепенный** трафик. Основной канал у успешных конкурентов (hubspeakers, topspeaker) — **сейлзы + бренд + ивент-нетворк**.

Все три аудита молча приняли SEO как главный канал. Это не обосновано в бизнес-кейсе.

### 5. **Поведенческие сигналы и Core Web Vitals не замерены**

SEO-RESEARCH.md упоминает «ПФ — фактор №1 у Яндекса», но **ни одного замера**: какой сейчас процент отказов? Глубина? Время на странице? Без этого все рекомендации — пальцем в небо.

Раздел «открытые вопросы» в ресёрче содержит эти вопросы, но они **не блокируют** план. Хотя должны.

### 6. **Коммерческие факторы Яндекса (для B2B критичны)**

Яндекс для коммерческих сайтов оценивает:
- Цены/прайсы на видном месте (есть, но без Schema)
- Физический адрес + карта (нет — `address` в LocalBusiness пустой)
- Реквизиты компании (ИП/ООО, ИНН) — обычно в подвале (статус не проверен)
- Раздел «О нас» / «Команда» — нет на сайте
- Отзывы клиентов — нет
- Сертификаты, лицензии — нет
- Гарантии, договор-оферта (есть? — PDF в подвале)

**Это сильнее влияет на коммерческую выдачу Яндекса, чем десяток Schema.org.**

### 7. **Email-индексация и спам-фильтры**

Если контакты спикеров видны в HTML — Яндекс/Google могут показывать в сниппете email спикера. Это значит: спикеры начнут получать спам сразу после индексации. Это **прямой жалоба** на сервис от платных пользователей.

### 8. **Mobile-first индексация и адаптивность**

Аудит ни словом не обмолвился про мобильную версию. С 2019 Google и Яндекс индексируют **мобильную** версию первой. Если на мобиле что-то схлопывается/скрывается через CSS (`hidden`, `lg:block`) — для бота этого нет.

Беглый просмотр `FirstScreen.js`, `PopularSpeakers.js` показывает массивные `hidden lg:block` / `hidden md:block`. Нужен проход по компонентам с фильтром Tailwind `hidden`.

### 9. **Технический долг в Strapi v5 миграции**

В коде встречаются обращения вида `data.categories.data.map(cat => cat.attributes?.Name)` — это **Strapi v4 API**. На Strapi 5 это **молча** возвращает undefined. То есть schema/маркап генерируются с битыми данными. Это не SEO-проблема, а технический долг, но он невидимо разрушает половину Schema.

## Альтернативные стратегии (помимо SEO-спринта)

### A. Прямые продажи 10 крупным ивент-агентствам и корпам

- 10 встреч × 2 часа = 20 часов
- Если 2 из 10 подписывают годовой контракт на размещение 20 спикеров (по 12 000 ₽/год за каждого) = **480 000 ₽/год гарантировано**.
- Сравнение: 4-недельный SEO-спринт (~120 часов) даст в лучшем случае рост органики на 30–50% за 3–6 месяцев → ~5–10 новых продаж/мес → ~30–60 тыс ₽/мес → ~360–720 тыс ₽/год **в успешном сценарии**.
- ROI **прямых продаж выше при меньших затратах часов**.

### B. Контентный маркетинг в Telegram (для B2B)

- Создать канал «Спикеры РФ» или партнёрство с существующими (например, «Event Manager», «HR-сообщество»).
- Размещать платные посты со спикерами.
- Telegram-каналы по событиям и HR имеют 5–50K подписчиков, ER 5–10%.
- Бюджет: 10–30 тыс ₽/пост.
- Это **точечный трафик от ЦА**, а не «жители Москвы случайно ищут спикера».

### C. Реферальная программа для самих спикеров

- Спикер приводит другого спикера → получает 30% от его подписки.
- Виральный рост базы без SEO.
- Затраты: разработка + механика учёта = 1–2 недели работы.
- Эффект — линейный к количеству существующих спикеров (нужна критическая масса).

### D. SEO в усечённой форме (минимальный must)

Не 4-недельный спринт, а **3 часа работы** с правильной приоритезацией (см. финальный вердикт).

### E. Партнёрство с конференциями

- Договориться с организаторами конференций (Russian Internet Week, Synergy, и т.д.) о том, чтобы их спикеры по умолчанию имели карточку на pluginexpert.
- Барт: вы → бесплатный год размещения для спикеров. Они → ссылка с сайта конференции и доверие к вашему бренду.
- Это даёт и ссылочный профиль (для SEO бонус), и контент (профили реальных топ-спикеров).

## Финальный вердикт: какие 3–5 действий действительно стоят первыми

Не «Неделя 1» из аудита, а **по реальной приоритетности с учётом ROI и риска**:

### 1. **Решить вопрос с защитой контактов (бизнес-критично, не SEO)**

Сначала ответьте на вопрос: «Если контакты в открытом HTML — почему спикер платит?». Без ответа бизнес-модель уязвима. **2 часа на обсуждение → решение → план**. Это блокер для всего остального — нет смысла гнать SEO-трафик на дырявый сервис.

### 2. **Перевести `BlogDetail.js` на серверный рендеринг**

Единственная по-настоящему CRITICAL SEO-задача проекта. 9 статей блога сейчас не индексируются по контенту. **1–2 часа работы**. Эталон уже есть в `(profile)/profile/[slug]/page.js`.

### 3. **Поставить verification-коды Яндекс.Вебмастер + Google Search Console + проверить какой sitemap.xml реально отдаётся**

- 30 минут на регистрацию и коды.
- `curl https://pluginexpert.ru/sitemap.xml` → понять, отдаётся ли статика или динамика.
- Удалить мёртвый файл из public/.
- Без Вебмастера дальнейшие действия — вслепую. Это не CRITICAL для трафика, но CRITICAL для измерений.

### 4. **Исправить 3 хирургических бага в коде:**
- `speaker.Photo.url` → `speaker.gallery?.[0]?.url` в `(profile)/profile/[slug]/page.js:29,75` (OG/Person image)
- `cat.attributes?.Name` → `cat.Name` (Strapi v5) в `(profile)/profile/[slug]/page.js:96` (knowsAbout)
- `notFound()` вместо `return <h1>Спикер не найден</h1>` (soft-404 → real 404)
- Бонус: `mailto:mailto:` в двух местах

**Всего ~1 час работы.** Точечные правки, измеримый эффект.

### 5. **Провести Wordstat-разведку перед категорийными страницами**

**2 часа** в Wordstat: проверить частотность 10 предполагаемых категорий. Если хотя бы 3 имеют >500 показов/мес — делать категорийные страницы. Если все ниже 100 — отказаться или отложить. **Не делать "потому что у конкурентов есть"** — это карго-культ.

### Что НЕ делать сейчас

- **FAQPage Schema** — мертва в Google с 7 мая 2026. Для Яндекса/AI — недоказанный эффект.
- **Полный набор Schema.org** (BlogPosting, ItemList, Service, ContactPoint, etc.) — низкий ROI, времязатратно. Сделать только BreadcrumbList + починить Person.
- **llms.txt** — согласен с агентами, отложить.
- **4-недельный спринт** в нынешнем виде — overscoped. Сделать 1 неделю по списку выше, замерить, потом решать.
- **Категории как первая задача** — без Wordstat-подтверждения.
- **SSR-перевод PopularSpeakers/ForthScreen/SpecialText одновременно** — это переписывание архитектуры, не «правка». Сделать только если шаг 1–5 закончены и есть данные из Вебмастера, что главная страница плохо ранжируется по содержательным запросам.

### Самый честный итог

**Самая важная задача — это не SEO, а ответ на вопрос «зачем спикеру платить, если контакты будут открыты?».** Без ответа любой SEO-успех = ускорение парсинга базы конкурентами. Аудиты дали хороший технический разбор, но **проглядели бизнес-модель**, которой они должны служить.
